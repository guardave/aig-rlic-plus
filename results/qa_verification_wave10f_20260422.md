## Findings Narrative

### F1–F2: indpro_xlp story + evidence — chart pending (ROOT CAUSE: deployment state mismatch)

**Symptom:** The Streamlit Cloud indpro_xlp Story and Evidence pages render "chart pending" with pair-prefix filenames (`indpro_xlp_hero.json`, `indpro_xlp_correlations.json`). These filenames do not exist in the current HEAD; they were git-renamed to bare names (`hero.json`, `correlations.json`) by commit `3c6bb50` (pushed 2026-04-22 08:30 UTC).

**Timing context:** The Wave 10F commits were pushed to `origin/main` at 08:30–08:36 UTC, only ~20 minutes before verification ran (08:50 UTC). Standard Streamlit Cloud redeploy cycle is 5–15 minutes, but the pair-prefix path strings in the fallback text are inconsistent with the current HEAD code.

**Root cause analysis:**

The fallback string "indpro_xlp_hero.json" can only be generated by one code path in `page_templates.py`:
```python
hero_chart = getattr(config, "HERO_CHART_NAME", f"{pair_id}_hero")
# fallback: f"output/charts/{pair_id}/plotly/{hero_chart}.json"
```
This produces "indpro_xlp_hero.json" only when `HERO_CHART_NAME` is absent or the default triggers. The current config at HEAD has `HERO_CHART_NAME = "hero"` (bare name). Two scenarios:
- (a) **Cloud is running bfb1b70 config** (where `HERO_CHART_NAME = "indpro_xlp_hero"`) because `3c6bb50` redeployed app code but the Streamlit instance cached a prior app state for the config module.
- (b) **Config import failure at runtime**: if `STORY_CONFIG` fails to import and the page falls back to a default config object, `HERO_CHART_NAME` would be absent → `getattr` default → "indpro_xlp_hero".

Notably, `indpro_xlp_strategy` **PASS**es with 5 charts — those charts (`equity_curves`, `walk_forward`, etc.) are bare-named in both bfb1b70 and HEAD configs, so they load from the renamed files. This suggests the cloud IS running current code for most paths, but STORY_CONFIG has a resolution issue.

**Contrast with HY-IG and UMCSENT:** Both pairs' story/evidence pages PASS cleanly. Their chart configs already used bare names before `3c6bb50`, so there is no ambiguity there. `indpro_xlp` is the only pair where the HERO/regime config fields changed from pair-prefix to bare at `3c6bb50`.

**Risk classification:** User-visible. Stakeholder opening indpro_xlp Story or Evidence sees 📊 "chart pending" placeholders instead of charts. High impact.

**Wave 10G item:** Investigate whether Cloud config import for indpro_xlp is loading the pre-`3c6bb50` version of STORY_CONFIG. One diagnostic: add a visible `st.caption(f"Hero chart name: {HERO_CHART_NAME}")` to the Story page temporarily. If it reads "indpro_xlp_hero", the issue is in the config import resolution (module cache or deployment state). If it reads "hero" but chart still shows pending, the chart file lookup is failing despite the correct name.
